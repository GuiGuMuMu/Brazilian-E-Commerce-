Non-negotiable guardrails

  • LLM cannot invent or calculate business metrics.
  • Every number in the insight must exist in Python output.
  • Every join step must log row counts before and after.
  • Every cleaning rule must be documented as an assumption.
  • Every alert remains a draft until human-approved.
This framing makes the class feel senior: the student learns control, auditability, and workflow design — not just syntax.

def inspect_table(df, name):
    return {
        "table":      name,
        "rows":       int(df.shape[0]),
        "columns":    int(df.shape[1]),
        "dtypes":     df.dtypes.astype("str").to_dict(),
        "missing":    df.isna().sum().astype(int).to_dict(),
        "duplicates": int(df.duplicated().sum()),
    }

table_summary = {name: inspect_table(df, name) for name, df in raw.items()}

03

Join / Manipulation Agent

Build an analysis-ready fact table at a stable grain without metric duplication.

+
PurposeTransform many raw tables into one analysis frame.
Inputorders, payments, items, reviews, customers.
OutputOrder-level fact table. Log row counts after each join.
Failure modeMany-to-many join that silently duplicates revenue rows.
Naive join on order_items before aggregating will create multiple rows per order and inflate revenue by item count.
# Aggregate BEFORE joining — never join at item grain to orders
items_agg = order_items.groupby("order_id").agg(
    item_count      = ("order_item_id",      "count"),
    product_revenue = ("price",              "sum"),
    freight_value   = ("freight_value",      "sum"),
    primary_category= ("product_category",
                       lambda x: x.mode().iloc[0]),
).reset_index()

fact = (
    orders
    .merge(customers,  on="customer_id", how="left")
    .merge(items_agg,  on="order_id",   how="left")
    .merge(payments,   on="order_id",   how="left")
    .merge(reviews[["order_id", "review_score"]],
                       on="order_id",   how="left")
)
# Always check null rates after joins
print(fact.isna().mean().round(3)[fact.isna().mean() > 0])

Data Quality Agent

Score data trust from 0–100 and document every cleaning assumption with business impact.

+
PurposeSurface trust issues before analysis. Make quality a measurable number.
InputRaw tables (before any cleaning).
OutputQuality score (0–100), issue list with business impact, cleaning rules.
Failure modeSilent cleaning with no business explanation or before/after comparison.
82 / 100 — example quality score
Deducted for duplicates, missing dates, negative delivery days 
def data_quality_agent(data):
    customers, orders, order_items, payments, reviews = (
        data[k].copy() for k in
        ["customers", "orders", "order_items", "payments", "reviews"]
    )
    orders["order_purchase_date"]  = pd.to_datetime(orders["order_purchase_date"],  errors="coerce")
    orders["order_delivered_date"]  = pd.to_datetime(orders["order_delivered_date"],  errors="coerce")
    delivery_days = (orders["order_delivered_date"] - orders["order_purchase_date"]).dt.days

    # --- Outlier detection (IQR method) ---
    q1, q3 = order_items["price"].quantile([.25, .75])
    price_outliers = int((order_items["price"] > q3 + 3 * (q3 - q1)).sum())

    # --- Referential integrity ---
    orphan_orders   = int((~orders["customer_id"].isin(customers["customer_id"])).sum())
    orphan_payments = int((~payments["order_id"].isin(orders["order_id"])).sum())

    issues = [
        {"category": "uniqueness",   "issue": "duplicate_customers",     "count": int(customers.duplicated().sum())},
        {"category": "completeness", "issue": "missing_age",              "count": int(customers["age"].isna().sum())},
        {"category": "completeness", "issue": "missing_delivery_date",     "count": int(orders["order_delivered_date"].isna().sum())},
        {"category": "validity",     "issue": "negative_delivery_days",   "count": int((delivery_days < 0).sum())},
        {"category": "validity",     "issue": "zero_or_negative_payments", "count": int((payments["payment_value"] <= 0).sum())},
        {"category": "outliers",     "issue": "price_outliers",           "count": price_outliers},
        {"category": "referential",  "issue": "orphan_records",           "count": orphan_orders + orphan_payments},
    ]
    score = 100 - sum(min(15, i["count"] * 2) for i in issues)
    return {"quality_score": max(0, score), "issues": issues}

+
PurposeProduce the numbers all downstream agents will reference.
InputClean fact table (delivered orders only).
OutputMonthly, channel, category-level metric tables.

delivered = fact[fact["order_status"] == "delivered"].copy()

monthly = delivered.groupby("order_month").agg(
    revenue         = ("payment_value",    "sum"),

EDA blueprint

A concrete diagnostic template — not "explore the data."

The student should leave knowing exactly which EDA outputs to produce, in what order, and what business question each one answers. Six blocks, six outputs.

EDA workflow

E1Data structure

Rows, dates, missingness, duplicates, distinct category values.

→
E2Metric sanity

Revenue distribution, outliers, zero/negative values, date gaps.

→
E3Time analysis

Monthly/weekly trends, MoM change, anomaly windows, change points.

→
E4Segment analysis

Revenue by channel, state, category. MoM change per segment. Top vs bottom.

→
E5Bivariate / relational

Delivery days vs review score. Revenue vs order count. Correlation matrix.

→
E6Hypotheses

Block 1 — Data structure & quality

  1. Table row counts and date coverage
  2. Missingness % by column
  3. Duplicate counts by natural key
  4. Distinct values in status/category fields
  5. Min/max dates and time gaps

Block 2 — Univariate metric sanity

  1. Distribution of payment_value
  2. Distribution of delivery_days_clean
  3. Distribution of review_score (1–5 bar)
  4. Top 1% order values — outlier check
  5. Zero / negative / null value audit

Block 3 — Time analysis

  1. Monthly revenue + MoM % change table
  2. Orders and customers over time
  3. AOV and avg review score over time
  4. Weekly trend if monthly looks flat
  5. Annotate anomaly windows

Block 4 — Segment analysis

  1. Revenue by channel / state / category
  2. MoM change by segment
  3. Top vs bottom segments
  4. New vs returning customer split
  5. Canceled / delayed orders by segment

Block 5 — Bivariate / relational

  1. Delay bucket vs avg review score
  2. Delay bucket vs low-review rate
  3. Payment value vs freight value
  4. Order count contribution curve
  5. Correlation matrix (numeric columns)

Block 6 — Customer behavior

  1. Repeat purchase rate overall
  2. Days between first and second order
  3. First-order-month cohort table
  4. Activity / email-open rate
  5. Churn proxy: no order within 90 days

Statistics / model

One model. Simple, interpretable, not trivial.

Logistic regression for low-review risk keeps the class practical while introducing proper statistical thinking: target definition, control variables, odds ratios, p-values, and the causation caveat.

C4
Predict Before You Run 2 min · before model runs
Before running the model — write your prediction. What sign and magnitude do you expect for each predictor?
delivery_days_clean OR will be:
Greater than 1 / Less than 1? ___
Significant (p<0.05)? ___
payment_value OR will be: ___
Direction: positive / negative? ___
Most surprising predictor: ___
💡 The prediction step is not about being right — it's about making the student an active participant in the model. When the output surprises their intuition (e.g., payment_value has a negative OR — higher-value orders get better reviews), that surprise is the lesson. It sticks because they made a stake.

Business framing

  • Question: Does longer delivery time increase the probability of a low review?
  • Target: low_review = 1 if review_score ≤ 2 else 0
  • Features: delivery_days_clean, payment_value, freight_value
  • Why logit: binary outcome, interpretable coefficients, business-relevant
import statsmodels.formula.api as smf
import numpy as np

model_df = order_fact[
    (order_fact["order_status"] == "delivered")
    & order_fact["delivery_days_clean"].notna()
    & order_fact["payment_value"].notna()
    & order_fact["freight_value"].notna()
].copy()

model = smf.logit(
    "low_review ~ delivery_days_clean + payment_value + freight_value",
    data=model_df
).fit(disp=False)

odds_ratios = np.exp(model.params).round(4)
p_values    = model.pvalues.round(4)
print(model.summary())

How to interpret — step by step

Coefficient sign
Positive → higher log-odds of low review. Negative → lower.
Odds ratio
OR = exp(coef). OR 1.08 on delivery_days means each extra day multiplies the odds of a bad review by 1.08 (+8%).
p-value
Evidence strength, not business importance. p < 0.05 is conventional. Always interpret alongside effect size.
Control variables
payment_value and freight_value control for order value before judging delivery days.
Causation caveat
Observational data. Unmeasured confounders exist. Do not say delivery causes bad reviews.
Student task: explain the delivery_days_clean odds ratio in one sentence to a VP of Operations. If they say "causes," correct it live. This is the most valuable 2 minutes of the class.

What makes an insight sentence defensible?

✗ Bad insight
"Revenue decreased this month."
A number without context is not an insight · No driver · No evidence · No action
✓ Defensible insight
"Revenue declined –14.3% in June (Python: monthly_metrics row 2024-06), driven by a –22% drop in Paid Search while Organic held flat (Python: channel_metrics). Avg delivery time rose from 5.1 to 6.8 days in the same period, coinciding with a 3-point increase in low-review rate."
Verified numbers · Source referenced · Driver named · Supporting evidence · No causal claim

Final deliverable

What the student should leave with.

1. Workflow fluency

The student can explain each agent's purpose, input/output contract, and failure mode without looking at notes.

2. EDA discipline

The student knows what 6 EDA outputs to produce for a revenue-diagnosis task, and can state the business question each one answers.

3. Statistical literacy

The student can explain a logistic regression odds ratio to a non-technical manager and correctly distinguish association from causation.

Student self-check — can you do these without notes?
  • Define revenue with grain, numerator, denominator, and filter before touching code.
  • Identify 5 categories of data quality issues and explain the business impact of each.
  • Build an order-level fact table from 5 raw tables without row duplication.
  • Compute monthly revenue, AOV, low-review rate, and MoM change in one groupby.
  • Run logistic regression and explain the delivery_days OR to a VP in plain English.
  • Produce a verified alert email draft where every number traces to a Python cell.
  • Explain in one sentence why you cannot send the email without human review.
If you teach this well, the student leaves knowing how a modern analytics workflow is designed — not just how to call groupby. The 2-year analyst who already knows pandas gains something they cannot get from documentation: the discipline to defend their numbers.
